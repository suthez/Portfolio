Project One

Task 1

Case Study

Our first task this semester was to write up a case study on a related subject to do with our Project. I chose a project that was undertaken by a group in New York/New Jersey. The people involved were a part of TheThingsNetwork New York Community. The project was based on smart gardens and utilizing these to help minimize the effects of extreme water events. Such as torrential rain, hurricanes and floods etc.

Task 3

Gateway/Raspberry Pi

To get ready for our gateway I had to clear the SD card and put a fresh installation of jesse lite onto it. The first step was to archive what the other group had done. This was followed by downloading a copy of jesse lite (lite because it was smaller in size) and writing it to the SD card. Task 4

Github Pages

I was tasked with managing our teams Github repository. This included archiving past teams work and implementing something different that was more flexible. For this we decided to use static Github pages run with Jekyll and formatted with Markdown language. Styling was made easy with several default layouts included when using Jekyll. There is a certain process to go through setting up Jekyll locally on your machine and is easier if you have certain operating systems. Basically everything but Windows. Windows does have support but it is limited and can have more issues. One specific problem I ran into while setting up was the way our pages were structured. We were using Github pages and not creating directly from the master repository, meaning some changes had to be made so the pages knew where to read the css and layouts from. Reflection

At the beginning of this project it seemed promising with the funds to back up what we thought would be a reliable solution. This however was soon to be discovered not to be the case. As this technology is still fairly new there is not a lot of support or information out there for a low budget solution. So our effots became more 'hacky' - trying to get older stuff to work and even our project manager buying his own hardware, although not in the right frequency. More research was done and we came to the decision to buy different hardware that seemed to be more promising, had a lot of documentation and support. While waiting for our new toy to arrive we managed to set up a working prototype with the gear we had - just in time for mini showcase! I found the showcase to be a good exercise as I had not had a lot to do with the setup of the server, gateway or node. As I was the only person from group avaliable at the time, I was responsible for moving the hardware and setting it up which in itself taught me a lot about how it worked. I think the groups efforts have produced a promising prototype that should be able to be transferred to the new hardware and on the right frequency. I definitely feel like we have made a good start and will have more of an idea of direction for next semester. Overall it was a slow start but looking back we have come a long way.

Project Two - Reflective Story

If you have checked out Project One you will discover it was done in a slightly different way. So improving from Project One I decided to change the layout and add a lot more detail. As Project One was done in a different layout/template I also decided to change and make improvements.

Picking up where we left off with Project 1: Again, maintaining the Projects Github website which, after all the changes, was renamed to IoT Dunedin. This was to to avoid all the hassle of renaming our Github repository - i.e breaking Jekyll links.

After a rough first semester I wanted to put more time into project. Having only two other classes, instead of three, the workload was a lot less which made it easier. So I decided to jump right in and get working on the project website. I started by going through and checking links. On discovery of broken links I went about fixing them. A task I didn't get around to implementing was a multi-drop-down navigation menu. This was a little difficult to implement as we were using a pre-made Jekyll theme (made for serving static Github pages). This theme did not implement navigation bars and therefore had no multi-drop-down functionality. Making your own custom css styles was fine but often resulted in clashes with the pre-made css and responsiveness. So while I could implement a drop-down functionality it would not fit the style when the screen size changed. I got frustrated with working on this and so took a break from it.

The next task was brought up by Campbell, to research and implement a privacy policy for end user. I thought this sounded like a fairly easy task, but as we discovered it was harder than we thought. Research resulted in sites that claimed you could easily create one using their website but then asking for payment at the end. It was difficult to determine what exactly we would need seeings as we did not even have a full plan for the user side of things. This task was soon set aside for a later date.

I decided to contribute to the research and implementation of server security. I started off by searching and documenting best practices for Ubuntu servers. Relevant information was saved in a Notepad file. This led me to setting a banner for the server. This was done using this website tutorial and this website to create the ASCII 'IoT Dunedin' banner. I followed the same site instructions as the banner for network hardening. This was to cover spoofing, block ping requests, and SYN attacks amongst other things. After implementing this, Oscar found he could no longer ping the server, so I had to undo the security I had implemented. During troubleshooting discovered I needed to restart the service for it to pick up the changes. I fixed this ASAP, straight after Oscar messaged me about the issue.

Patrick also started working on server security. He took this on more as his main task than I did, so I gave him my notes and he implemented majority of the security.

A group decision saw our Github repository move to a new location to start fresh, as we decided to change the name. The reason for this was to avoid the headache of having to make changes on all pages where the repository name was and other errors. This gave us a new url and instead of 'TheThingsNetwork' it was now 'DunedinIoT'. We then discovered Google still had the old link available on a search, which would then lead to a 404. To remedy this I first did a quick search and found you could request Google to remove a dead link from coming up in a search.

Back to working on the website, I decided to add a favicon to replace the default and boring page looking icon. I used this free favicon site and chose a bright, appealing icon. I also continued working on fixing the multi-drop-down menu. This still was proving hard to fix and so a group decision was made to find a new theme. It was recommended to use a Bootstrap theme which was responsive.
Bootstrap has many useful functionalities that are fairly easy to implement which was a bonus. Team member Nick found a one-page Bootstrap template that was quite a nice style and looked professional. Being a one-page website meant that there were no links to other pages merely a link to an anchor in the page, scrolling automatically to the anchor. This posed a problem as we had many different pages and thought this wouldn't be suitable. However, using Jekyll meant that we could create different layouts. So we created a new layout to use for pages other than the Homepage.

As this semester had new members joining the group, I had two extra people working on the website. This made it a little difficult to decide what each member was doing and sometimes resulted in overlapping - people working on the same thing. As these guys were new to static Github pages it was my role to show them how to set it up so you could locally build and test. The benefit of this was to be able to make changes locally, test,view and then push the desired changed to Github. This reduced the amount of commits made to Github.
Using Jekyll with Windows could result in errors when trying to run or build from a different location. Often we would have to troubleshoot to get going again with these steps or, if failing this, pulling the repository again from Github.

Nick found and put up a picture of the Dunedin Railway Station as our banner for homepage. It was brought up that this would probably need permission or at least a reference to the picture. I took this on and started researching about licensing and copyright. Luckily a search for this picture brought me to the Wikipedia site of the author which had a CC (Creative Commons) license. I used the creative commons website to get the exact syntax for giving attribution.

The next task I started on was implementing a breadcrumb navigation. This was because our site was quite the labyrinth when navigating and there was no easy way to get around the site. Breadcrumbs gives you a trail so you can easily return to previous pages or the Home page. This seemed easy enough apart from the fact that what I was looking for was a dynamic breadcrumb menu. Given the fact I did not fully understand Jekyll and liquid tags, finding a solution was difficult. I finally found a solution which was to add a file and make sure to include it on every page. The website that contains the link of the file is Jekyll Codex. Once the breadcrumb nav was implemented, I started working on the showcase website. I started off by implementing Jekyll so I could locally build. I then researched about mqtt dashboards, but more specifically a Github solution. It was soon found that we were having problems with static Github and it not being able to make requests. We then decided to host our showcase website on our server. This bypassed the issues we were experiencing. After a lot of research I came across a helpful Github repository. This site seemed to have what we were looking for. From here, I went on to discover where the author of the Github site got the gauge from. It was here that I found a lot of good documentation for the gauge such as changing the range colours, ticks and label formatting.

The next step was to connect to mqtt. For this I spoke to Oscar who had knowledge of this. After much testing soon I was connected, subscribed and receiving messages from the server! I wrote up some documentation on this which you can check out here.

For the second last week there were some last minute touch ups for the showcase website, followed by setting up for showcase night. On the night of showcase we got the chance to interact with the public and show off our accomplishments. In the end we had a working gateway, server, nodes, and webapp connecting and subscribing to the server. Also we had done some range testing of a node to the gateway and made an impressive map which covered a lot of Dunedin. We were happily surprised by the range which was around 6kms when we expected only 2-2.5 kms.

The last week will be spent finishing up the Github repository, getting ready to hand over to the next year's project students, and writing up our group technical report.

Conclusion

In conclusion, I have found this particular project to be a challenge. With the technology being so new and the concept not being greatly documented it was hard to know what to do at times. Often I had issues of being stuck and lacking in motivation. This possibly resulted in not getting as much work done as I could have. Also, it was hard to share tasks as there wasn't a whole lot to do. At these times I would talk to Tom our project manager and find ways to contribute or get ideas when I had none. This was very helpful and I always came away feeling refreshed and ready to work.

I feel our team did a good job considering the state of the project when we started. It has been long and arduous but the end result is quite satisfying and exciting for the future.

Overall, I enjoyed being a part of something so new and different. I feel that with such an interest in IoT and LoRA networks it has given me skills and knowledge that others may not have. I feel I have progressed and am ready to graduate and start a career in IT.
